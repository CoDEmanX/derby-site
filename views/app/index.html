Client and Server rendering

On first request, html is rendered server-side, which is good for page loading time and SEO. On subsequent url changes, html is rendered client-side, which is good for interactivity.

Why not Rails, Django, Yii, Asp Net MVC?

Adding dynamic features to apps written with server-side frameworks tends to produce a tangled mess. Server code renders various initial states while jQuery selectors and callbacks desperately attempt to make sense of the DOM and user events. Adding new features typically involves changing both server and client code, often in different languages.

Why not Angular, Backbone, Marionette, Knockout, Ember?

Client-side MVC frameworks helps to better structure client code. Declarative model-view binding libraries reduces boilerplate DOM manipulation and event bindings. These are great concepts, and adding some structure certainly improves client code. However, they still lead to duplicating rendering code and manually synchronizing changes in increasingly complex server and client code bases. Not only that, each of these pieces must be manually wired together and packaged for the client. And client-side html rendering only is not good for SEO and page loading time.

Why not Meteor?

Meteor has very nice api and you can make your app in 'just two clicks'. That`s great for simple apps. Price for this is monolithic framework structure. The more your app is growing, the more flexibility you want from framework. That`s where Derby comes to play.

Why Derby?

The only isomorphic framework. The only framework with OT conflict resolution. It`s scallable, flexible (npm, express, etc) without glue code. Derby is good for writing all kinds of web-apps from static web-sites (even without database) to realtime multiplayer games and collaborative applications. Derby is Awesome.
